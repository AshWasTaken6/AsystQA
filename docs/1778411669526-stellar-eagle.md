# Backend Architectural Overhaul: Reactive → Proactive System Transformation

## AsystQA — Strategic Roadmap for Observability, Investigation & Reliability

---

## Executive Summary

AsystQA's current backend is a synchronous, single-process FastAPI application with four heuristic-based code-analysis agents, a plaintext JSON data store, zero authentication, no observability infrastructure, and no automated testing. This plan transforms the system into a proactive, observable, resilient platform through six phases: Audit & Gap Analysis, Observability & Investigation Framework, Architectural Modernization, Data Integrity & Security, DevOps & Reliability Engineering, and a phased Execution Roadmap.

---

## 1. Audit and Gap Analysis

### 1.1 Current State Inventory

| Domain | Current State | Gap |
|--------|--------------|-----|
| **Logging** | Basic `logging.basicConfig` with format strings | No structured logging, no correlation IDs, no log levels per module |
| **Metrics** | None | No Prometheus metrics, no SLI/SLO tracking |
| **Tracing** | None | No distributed tracing, no request lifecycle visibility |
| **Monitoring** | None | No dashboards, no alerting, no anomaly detection |
| **Error Handling** | Unhandled exceptions bubble as 500s | No global exception handler, no error classification |
| **Health Checks** | `GET /` returns `{"status": "..."}` only | No `/healthz`, `/readyz`, `/livez` endpoints |
| **Authentication** | None | Open API, no RBAC, no token validation |
| **Data Store** | Plain `memory/history.json` | No encryption, no backups, no integrity checks, no concurrent-write safety |
| **Testing** | Zero test files | No unit, integration, or E2E tests |
| **CI/CD** | None | No pipeline, no automated deployment |
| **Resilience** | None | No circuit breakers, no retries, no rate limiting, no graceful degradation |
| **API Versioning** | None | Single `/analyze` endpoint, no version prefix |
| **Agent Quality** | Keyword-based heuristics | No context awareness, no ML, no confidence scoring |
| **Request Tracing** | None | Cannot trace a single request across agents |
| **Documentation** | Minimal | No OpenAPI spec generated, no runbooks |

### 1.2 Blind Spots Identified

1. **No request lifecycle visibility** — A code submission enters `/analyze` and produces a result, but there is no way to trace what happened inside the pipeline, which agent took how long, or where failures occur.
2. **No failure taxonomy** — Errors are not classified (transient vs. permanent, client vs. server, data vs. logic).
3. **No performance baseline** — No knowledge of what "normal" latency, memory usage, or throughput looks like.
4. **No security audit trail** — No record of who submitted what code, when, or from where.
5. **No data integrity verification** — History JSON can be corrupted, overwritten, or tampered with silently.
6. **No proactive alerting** — Problems are discovered only when users report them.

### 1.3 Gap Analysis Methodology

- **Static analysis** of all Python files for error handling coverage, type annotation completeness, and security anti-patterns.
- **Load test** baseline: measure p50/p95/p99 latency, throughput ceiling, memory usage under concurrent requests.
- **Failure injection**: Simulate agent failures, memory corruption, slow upstream calls to understand cascading behavior.
- **Telemetry audit**: Enumerate every data point the system produces and map it against the observability pillars (metrics, logs, traces).

## 2. Observability & Investigative Framework

### 2.1 Structured Logging

- **Replace** `logging.basicConfig` with `structlog` (already referenced in the Zero Trust security plan).
- **Log schema** (every log line is JSON):
  ```json
  {
    "timestamp": "ISO-8601",
    "level": "INFO|WARNING|ERROR|DEBUG",
    "logger": "module.name",
    "correlation_id": "uuidv4",
    "event": "descriptive_event_name",
    "context": { "key": "value" }
  }
  ```
- **Propagate** `correlation_id` via FastAPI middleware — generate on inbound request, attach to all downstream log calls.
- **Log levels per module**: `pipeline=INFO`, `agents=DEBUG`, `api=INFO`, `memory=WARNING`, `core=INFO`.

### 2.2 Metrics (Prometheus)

- **Integrate** `prometheus-client` (already in security plan dependencies).
- **Expose** `/metrics` endpoint (standard Prometheus scrape target).
- **Define core metrics**:
  - `asystqa_request_duration_seconds` (Histogram, labels: endpoint, status_code, language)
  - `asystqa_pipeline_agent_duration_seconds` (Histogram, labels: agent_name)
  - `asystqa_scan_total` (Counter, labels: language, status)
  - `asystqa_errors_total` (Counter, labels: error_type, module)
  - `asystqa_memory_file_size_bytes` (Gauge)
  - `asystqa_queue_depth` (Gauge — pending scans if async)

### 2.3 Distributed Tracing (OpenTelemetry)

- **Integrate** `opentelemetry-sdk` + `opentelemetry-instrumentation-fastapi` + `opentelemetry-exporter-otlp`.
- **Trace context propagation**: Every `/analyze` request generates a trace spanning:
  ```
  Gateway → Analyze Endpoint → Planner Agent → Reviewer Agent → Security Agent → Tester Agent → Reporter → Memory Update
  ```
- **Each agent span** records: agent name, input size, output count, duration, errors.
- **Trace ID = correlation ID** — unified across logs, metrics, and traces.
- **Export to**: Grafana Tempo / Jaeger / Zipkin (configurable via env var).

### 2.4 Real-Time Error Profiling

- **Global exception handler** in FastAPI middleware:
  - Captures unhandled exceptions, logs full traceback with correlation ID.
  - Increments `asystqa_errors_total` with error classification.
  - Returns standardized error response with request ID for customer support.
- **Sentry integration** (optional): `sentry-sdk` for real-time error alerting with stack traces.
- **Error budget tracking**: Define SLOs (e.g., 99.9% success rate) and track burn rate.

### 2.5 Health Check Endpoints

```
GET  /healthz         — Liveness probe    (is process running?)
GET  /readyz          — Readiness probe   (can serve requests? dependencies OK?)
GET  /livez           — Deep liveness     (can write to memory store?)
GET  /startupz        — Startup probe     (has initialization completed?)
```

### 2.6 Dashboard & Alerting

- **Grafana dashboards** with panels for:
  - Request rate, latency percentiles, error rate
  - Per-agent processing time breakdown
  - Top error types and affected languages
  - Memory store growth rate
  - Scan volume over time
- **Alerting rules** (via Prometheus Alertmanager or Grafana Alerts):
  - Error rate > 5% over 5 minutes → PagerDuty/Slack P1
  - P99 latency > 10s → P2
  - Memory store file size > 100MB → P3
  - No successful scans in 30 minutes → P2

## 3. Architectural Modernization

### 3.1 Async Pipeline

- **Migrate** `run_pipeline()` from synchronous to `async` using `asyncio.gather()` for independent agent calls.
- **Agents** become async-aware: `async def run_reviewer(...)`, etc.
- **Benefit**: Parallel agent execution reduces end-to-end latency from sum-of-times to max-of-times.

### 3.2 Event-Driven Architecture (Incremental)

- **Phase A**: Introduce a task queue (Celery + Redis or Dramatiq) for scan processing.
  - `/analyze` enqueues a task and returns a `scan_id` immediately (202 Accepted).
  - Client polls `/results/{scan_id}` or receives a webhook.
- **Phase B**: Agent communication via message broker (Redis Streams / RabbitMQ).
  - Pipeline orchestrator dispatches to agents as independent workers.
  - Enables horizontal scaling of individual agents.
- **Phase C** (Future): Event sourcing for audit trail — every state change is an event.

### 3.3 API Versioning

- **URL-based versioning**: All routes prefixed with `/v1/`.
  ```
  POST /v1/analyze
  GET  /v1/history
  GET  /v1/metrics
  GET  /v1/healthz
  ```
- **Version negotiation**: Accept header `Accept: application/vnd.asystqa.v2+json` for future versions.
- **Deprecation policy**: Each version supported for 6 months after new version GA.

### 3.4 Agent Architecture Improvements

- **Agent registry**: Central agent registry with agent name, version, health status.
- **Confidence scoring**: Each agent finding includes a confidence level (low/medium/high).
- **Context enrichment**: Agents receive not just code+language but also pipeline context (previous agent results, project metadata).
- **Pluggable agent interface**: Abstract base class `BaseAgent` with `analyze(code, context) -> AgentResult` — enables adding new agents without modifying pipeline.

### 3.5 Response Schema Enrichment

- **Add to `AnalyzeResponse`**:
  - `scan_id: UUID` — unique identifier for tracing
  - `correlation_id: str` — for log/trace correlation
  - `agent_timings: Dict[str, float]` — per-agent processing time
  - `confidence: float` — overall report confidence score
  - `warnings: List[Dict]` — non-blocking warnings from agents

## 4. Data Integrity & Security

### 4.1 Authentication & Authorization

- **OAuth2/OIDC** integration (Auth0, Keycloak, or self-hosted):
  - `POST /auth/token` — token endpoint (client_credentials for API, password for users)
  - `POST /auth/register` — user registration
  - Token stored in HTTP-only secure cookies.
- **RBAC model**:
  - `viewer` — can submit scans, read own results
  - `analyst` — can read all history, export reports
  - `admin` — can manage users, view audit logs, configure system
- **Route protection**: FastAPI `Depends()` with OAuth2 scheme on all routes except `/healthz`, `/docs`, `/metrics`.

### 4.2 Encryption

- **At rest**: Encrypt `memory/history.json` using AES-256-GCM via `cryptography` library.
  - Create `backend/services/encryption.py` with encrypt/decrypt functions.
  - Key managed via environment variable (development) or Vault (production).
- **In transit**: Enforce TLS 1.3 via nginx reverse proxy or cloud load balancer.
- **Secret redaction**: Before storing any scan results, detect and redact potential secrets (API keys, tokens) using regex patterns + `presidio-analyzer`.

### 4.3 Audit Trail

- **Create `backend/services/audit.py`**:
  - Every request logged to `audit/history.json` (separate from scan history).
  - Audit log schema:
    ```json
    {
      "timestamp": "ISO-8601",
      "correlation_id": "uuid",
      "user_id": "str|null",
      "action": "scan.submitted|scan.completed|login|logout|config.changed",
      "resource": "scan|user|config",
      "resource_id": "str|null",
      "outcome": "success|failure",
      "ip_address": "str",
      "metadata": {}
    }
    ```
- **Immutable append-only**: Audit logs are append-only; old entries never modified.

### 4.4 Data Validation & Integrity

- **Request validation**: Expand `AnalyzeRequest` schema:
  - `code`: max_length=1_000_000 (1MB limit)
  - `language`: validate against known language list
  - `filename`: optional, sanitized
- **Response validation**: Pydantic validators on output schemas.
- **Integrity checks**: SHA-256 hash of history.json, verified on load.
- **Backup**: Automated daily backup of `memory/history.json` → `memory/backups/`.

### 4.5 Rate Limiting

- **Middleware-based**: `slowapi` or custom middleware.
- **Limits**:
  - Anonymous: 10 requests/minute
  - Authenticated free tier: 60 requests/minute
  - Authenticated premium: 600 requests/minute
  - Admin: 1000 requests/minute

## 5. DevOps & Reliability Engineering

### 5.1 Testing Strategy

**Unit Tests** (`tests/unit/`):
- Each agent tested individually with known code samples.
- Mock external dependencies.
- Target: 80%+ coverage on agent logic.

**Integration Tests** (`tests/integration/`):
- Full pipeline execution via TestClient.
- Test schema validation (valid/invalid inputs).
- Test memory read/write cycle.
- Test authentication flows.

**End-to-End Tests** (`tests/e2e/`):
- Full scan lifecycle: submit → poll result → verify report.
- Concurrent scan handling.
- Error recovery scenarios.

**Test data fixtures**: `tests/fixtures/` with sample code files per language.

**Framework**: `pytest` + `httpx` (async test client) + `pytest-asyncio`.

### 5.2 CI/CD Pipeline

```yaml
# .github/workflows/ci.yml
stages:
  1. lint (ruff + mypy)
  2. unit tests (pytest)
  3. integration tests
  4. security scan (bandit, safety)
  5. build (uv + Docker)
  6. deploy to staging
  7. e2e tests
  8. deploy to production (manual approval gate)
```

### 5.3 Self-Healing Mechanisms

**Circuit Breaker** (per agent):
- Use `pybreaker` or custom implementation.
- If an agent fails 5 times in 60 seconds → open circuit → return fallback response.
- Half-open state after 30 seconds → test with one request.

**Retry with Exponential Backoff**:
- Failed agent calls retry up to 3 times with backoff (1s, 2s, 4s).
- Jitter added to prevent thundering herd.

**Graceful Degradation**:
- If a single agent fails, pipeline continues with remaining agents.
- Report indicates which agents succeeded/failed.
- If all agents fail, return 503 with retry-after header.

**Timeout Enforcement**:
- Each agent has a 10-second timeout (configurable).
- Entire pipeline has a 45-second timeout.
- Timeout exceeded → partial results with warning.

### 5.4 Containerization

- **Dockerfile**: Multi-stage build (builder + runtime).
- **docker-compose.yml**: App + Redis (task queue) + Prometheus + Grafana.
- **Non-root execution**: Container runs as non-root user.
- **Read-only filesystem** where possible.

### 5.5 Configuration Management

- **Environment variables** for all config (no hardcoded values).
- `.env.example` file with all available options.
- **Secrets**: Never in environment variables in production — use Vault or cloud secrets manager.

## 6. Execution Roadmap

### Phase 0: Foundation (Weeks 1–2)

| Task | Deliverable |
|------|------------|
| Global exception handler middleware | Unhandled errors return structured JSON with correlation ID |
| Correlation ID middleware | Every request gets a traceable ID propagated to all logs |
| Structured logging (`structlog`) | All log output is JSON with timestamp, level, module, correlation_id |
| Unit test scaffolding (`pytest` config) | `tests/` directory with conftest and first unit tests |
| Expand request validation | `AnalyzeRequest` with max_length, language whitelist, filename |
| Error classification schema | Categorize all possible errors (client 4xx, server 5xx, data, auth) |

**Exit Criteria**: Every request has a correlation ID that appears in all logs. All unhandled errors return structured responses.

### Phase 1: Observability Core (Weeks 3–4)

| Task | Deliverable |
|------|------------|
| Prometheus metrics integration | `/metrics` endpoint exposing request duration, error counts, scan counters |
| OpenTelemetry tracing | Full trace from `/analyze` through all agents with span timing |
| Grafana dashboards | Request rate, latency, errors, per-agent timing |
| Alerting rules | PagerDuty/Slack alerts for error rate > 5%, latency > 10s p99 |
| Health check endpoints | `/healthz`, `/readyz`, `/livez`, `/startupz` |
| Agent timing instrumentation | Each agent reports its processing duration |

**Exit Criteria**: On any request, you can search by correlation_id and find its complete trace across logs, metrics, and traces. Dashboards are live.

### Phase 2: Resilience & Security (Weeks 5–7)

| Task | Deliverable |
|------|------------|
| Circuit breaker per agent | Failed agent degrades gracefully, report indicates failure |
| Retry with exponential backoff | Transient failures auto-retried up to 3x with jitter |
| Rate limiting middleware | Per-tier rate limits enforced |
| OAuth2/OIDC authentication | `/auth/token` endpoint, protected routes |
| RBAC implementation | `viewer`, `analyst`, `admin` roles with permissions |
| AES-256 encryption at rest | `memory/history.json` encrypted |
| Secret redaction pipeline | Detected secrets in code are redacted before storage |
| Audit logging | Append-only audit log with user, action, resource, outcome |
| API versioning | All routes under `/v1/` prefix |
| Async pipeline | Agents execute concurrently via `asyncio.gather()` |

**Exit Criteria**: All routes require authentication. Encrypted history file is read/written correctly. Simulated agent failure produces graceful degradation with partial results.

### Phase 3: Data & Reliability Hardening (Weeks 8–9)

| Task | Deliverable |
|------|------------|
| Daily backup system | Automated backups of history + audit logs |
| Data integrity verification | SHA-256 checksum validation on load |
| Integration test suite | Full pipeline tested via TestClient |
| E2E test suite | Scan lifecycle tested end-to-end |
| Bandit security scan | No critical/high security findings |
| Memory store migration path | Script to migrate from plaintext to encrypted format |

**Exit Criteria**: All tests pass in CI. Backup/restore procedure tested. Data integrity checks catch corruption.

### Phase 4: Event-Driven & Scale (Weeks 10–12)

| Task | Deliverable |
|------|------------|
| Task queue (Celery/Dramatiq + Redis) | Scan processing is asynchronous with task ID |
| Agent workers as separate processes | Each agent runs as independent worker |
| Poll-based results endpoint | `GET /v1/results/{scan_id}` |
| WebSocket support (optional) | Real-time scan progress updates |
| Docker Compose deployment | Full stack runs via `docker-compose up` |
| Non-root container execution | Production container runs as non-root |
| Load test | Validated to defined throughput targets (e.g., 50 concurrent scans) |
| Documentation | API docs (auto-generated OpenAPI), runbooks, architecture decision records |

**Exit Criteria**: Horizontal scaling verified by load test. Full deployment via `docker-compose up`. API documentation auto-generated.

### Phase 5: Continuous Improvement (Ongoing)

| Task | Frequency |
|------|-----------|

| Load test baseline comparison | Monthly |
| Error budget review | Monthly |

## Cross-Cutting Concerns

### Developer Experience
- All configuration via `.env` with `.env.example` template.
- `make dev` starts full development environment.
- `make test` runs full test suite.
- `make lint` runs ruff + mypy.
- Auto-generated API docs at `/docs` (Swagger UI).

### Migration Strategy
- **Parallel operation**: Old synchronous pipeline runs alongside new async pipeline during transition.
- **Feature flags**: New features behind feature flags for rollback capability.
- **Blue-green deployment**: Zero-downtime deployment strategy.
- **Data migration**: Encrypted history file with migration script that reads plaintext, encrypts, and replaces.

### Measurable Success Criteria

| Metric | Current | Target |
|--------|---------|--------|
| MTTD (Mean Time to Detect) | Infinite (no monitoring) | < 5 minutes |
| MTTR (Mean Time to Resolve) | Infinite (no tracing) | < 30 minutes |
| Test coverage | 0% | > 80% |
| Error rate visibility | None | 100% classified |
| Authentication coverage | 0% | 100% of routes |
| Data encryption | None | 100% at rest |
| Pipeline latency | ~5s (sequential) | ~2s (parallel) |
| Deployment time | Manual | < 10 min automated |
