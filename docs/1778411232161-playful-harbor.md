# Zero Trust Security Architecture Implementation Plan
## AsystQA Multi-Layered Internal Data Security System

---

## 1. EXECUTIVE SUMMARY

**Objective:** Transform AsystQA from a basic code analysis service into a Zero Trust-secured platform with defense-in-depth controls protecting sensitive data at rest, in transit, and in use.

**Scope:** Backend API service, data storage, agent communications, and future scaling considerations.

**Timeline:** 5 implementation phases across ~6-8 weeks.

**Key Principles:**
- Never trust, always verify
- Least privilege access
- Assume breach mindset
- Explicit authentication/authorization for every request
- Comprehensive audit trails
- Automated threat detection

---

## 2. CURRENT STATE ASSESSMENT

### Existing Security Posture
- **Authentication:** None (open API endpoints)
- **Authorization:** None (all users have full access)
- **Encryption:** No at-rest encryption, no enforced TLS
- **Audit:** Basic timestamp logging in history.json
- **Monitoring:** None beyond simple counters
- **Secrets Management:** None (hardcoded configs, no .env)
- **Network Security:** None (single service, no segmentation)
- **Data Integrity:** None (plain JSON file storage)

### Identified Risk Areas
1. **Unauthorized Access:** Anyone can submit code or retrieve analysis history
2. **Data Leakage:** Submitted code (may contain API keys, passwords) stored unencrypted
3. **Insider Threat:** No separation of duties, no privileged access controls
4. **Compliance Gap:** No audit trails for regulatory requirements
5. **Tampering:** No integrity verification on stored data or logs

## 4. DETAILED LAYERED IMPLEMENTATION PLAN

---

### **LAYER 1: IDENTITY AND ACCESS MANAGEMENT (IAM)**

#### Goals
- Strong authentication for all users, services, and admins
- Role-based access control (RBAC) with least privilege
- Session management and token lifecycle controls

#### Implementation Steps

**Phase 1A: Authentication Foundation**
- [ ] Add `authlib` or `python-jose` + `passlib` to requirements.txt
- [ ] Create `backend/core/auth.py` with OAuth2/OpenID Connect support
- [ ] Implement JWT-based authentication with RS256 signatures
- [ ] Configure OAuth2 provider (Auth0, Okta, or self-hosted Keycloak)
- [ ] Add `/auth/login`, `/auth/logout`, `/auth/refresh` endpoints
- [ ] Integrate middleware to protect all routes (except public docs/health)

**Phase 1B: Multi-Factor Authentication (MFA)**
- [ ] Support TOTP via `pyotp` (Google Authenticator)
- [ ] Add WebAuthn/FIDO2 hardware security key support via `webauthn` library
- [ ] Implement biometric fallback (platform authenticators)
- [ ] Enforce MFA for admin endpoints (`/admin/*`, `/history`, `/config`)
- [ ] MFA enrollment flow with recovery codes
- [ ] MFA challenge-response for sensitive operations

**Phase 1C: Authorization & RBAC**
- [ ] Define roles: `viewer`, `analyst`, `admin`, `service_account`
- [ ] Create `backend/core/authorization.py` with permission checks
- [ ] Implement granular permissions:
  - `code:analyze` - Submit code for analysis
  - `history:read` - View analysis history
  - `history:delete` - Purge sensitive analyses
  - `config:read` - View system configuration
  - `config:write` - Modify system settings
- [ ] Add `@require_permission` decorator for route protection
- [ ] Enforce least privilege: analysts cannot access admin configs

**Phase 1D: Service-to-Service Authentication**
- [ ] Create machine-to-machine (M2M) credentials
- [ ] Mutual TLS (mTLS) for inter-agent communication
- [ ] Service identity via signed JWTs with audience claims
- [ ] Rotate service keys quarterly via automated pipeline

### **LAYER 2: DEFENSE-IN-DEPTH ARCHITECTURE**

#### Goals
- Multiple concentric security layers
- Network segmentation and isolation
- Advanced endpoint security
- Encrypted communications throughout

#### Implementation Steps

**Phase 2A: Network Micro-segmentation**
- [ ] Deploy each agent as separate microservice (Docker containers)
- [ ] Use Docker network policies to restrict inter-service traffic
- [ ] Segment: `frontend`, `api-gateway`, `agents`, `database`, `logging`
- [ ] Implement sidecar proxy (Envoy/Linkerd) for mutual TLS
- [ ] Enable egress filtering (agents cannot call external APIs without approval)
- [ ] Create bastion host for admin SSH access only

**Phase 2B: API Gateway & Edge Security**
- [ ] Deploy Kong/Traefik as API gateway:
  - Rate limiting (100 req/min per user/IP)
  - Request size limits (max 1MB code submission)
  - Request validation and sanitization
  - DDoS protection via cloud provider WAF
- [ ] Configure TLS everywhere (no HTTP allowed)
- [ ] Harden headers: `Strict-Transport-Security`, `X-Frame-Options`
- [ ] Enable CORS with specific origins only (remove wildcard)

**Phase 2C: Endpoint Detection & Response (EDR)**
- [ ] Install Falco (kernel-level security) on host
- [ ] Configure audit rules for:
  - File access to sensitive directories (`/keys`, `/secrets`)
  - Privileged process spawns
  - Unusual network connections
- [ ] Deploy OSSEC/Wazuh for host-based IDS
- [ ] Enable file integrity monitoring (FIM) on config files
- [ ] Integrate with alerting (email/Slack for immediate threats)

**Phase 2D: Secure Inter-Service Communication**
- [ ] Encrypt all internal traffic with TLS 1.3
- [ ] Implement mTLS between services (service mesh)
- [ ] Use SPIFFE/SPIRE for service identity (optional, advanced)
- [ ] Validate certificates with short TTL (24h auto-rotation)
- [ ] Monitor for certificate expiry with automated renewals

**Phase 2E: Runtime Protection**
- [ ] Enable seccomp/AppArmor profiles for Python processes
- [ ] Run containers as non-root users
- [ ] Implement read-only filesystem for code execution
- [ ] Add resource limits (CPU/memory) per agent container
- [ ] Enable Docker content trust (image signing)

### **LAYER 3: DATA PROTECTION & ENCRYPTION**

#### Goals
- End-to-end encryption of all sensitive data
- Strong encryption at rest with proper key management
- Data masking and tokenization for PII/secrets

#### Implementation Steps

**Phase 3A: Encryption Standards & Key Management**
- [ ] Choose AWS KMS, HashiCorp Vault, or Azure Key Vault for key storage
- [ ] Generate AES-256 keys for data at rest (one per data classification level)
- [ ] Implement envelope encryption for data:
  - Generate unique data encryption key (DEK) per analysis
  - Encrypt DEK with master key (KEK) from KMS
  - Store encrypted DEK alongside encrypted data
- [ ] Rotate master keys annually, DEKs per analysis
- [ ] Enable key versioning and automated key rotation

**Phase 3B: Encryption at Rest**
- [ ] Encrypt `memory/history.json` using AES-256-GCM
- [ ] Create `backend/services/encryption.py`:
  - `encrypt_data(plaintext: str, key_id: str) -> dict`
  - `decrypt_data(ciphertext: dict, key_id: str) -> str`
- [ ] Migrate existing history.json to encrypted format:
  - Read plaintext, encrypt, replace file, log migration
- [ ] Encrypt log files containing sensitive data
- [ ] Enable filesystem-level encryption (LUKS/dm-crypt) on storage volumes

**Phase 3C: Encryption in Transit**
- [ ] Enforce HTTPS-only with TLS 1.3 minimum
- [ ] Configure strong cipher suites only:
  - `TLS_AES_256_GCM_SHA384`
  - `TLS_CHACHA20_POLY1305_SHA256`
  - Disable RSA key exchange, use ECDHE only
- [ ] Enable Perfect Forward Secrecy (PFS)
- [ ] Use certificates from trusted CA (not self-signed in production)
- [ ] Implement certificate pinning for high-value clients (optional)
- [ ] Set up automatic certificate renewal (Let's Encrypt or ACM)

**Phase 3D: Data Masking & Tokenization**
- [ ] Implement secret redaction before storage:
  - Detect patterns: API keys, passwords, tokens, private keys
  - Replace with `[REDACTED]` or tokenized placeholder
- [ ] Use Python `presidio` or custom regex for PII detection
- [ ] Store original secrets only in isolated, encrypted vault
- [ ] Create `backend/services/redaction.py`:
  - `redact_secrets(code: str) -> tuple[str, dict]` (redacted_code, secret_map)
  - `restore_secrets(redacted: str, secret_map: dict, user_id: str) -> str`
- [ ] Access to original secrets requires MFA + audit log entry

**Phase 3E: Database Hardening** (Future)
- [ ] Migrate from JSON to PostgreSQL with pgcrypto
- [ ] Enable Transparent Data Encryption (TDE)
- [ ] Column-level encryption for highly sensitive fields
- [ ] Row-Level Security (RLS) for multi-tenancy isolation

### **LAYER 4: CONTINUOUS MONITORING & BEHAVIORAL ANALYTICS**

#### Goals
- Detect anomalies in user and system behavior
- Real-time threat intelligence
- Automated response to suspicious activity

#### Implementation Steps

**Phase 4A: User and Entity Behavior Analytics (UEBA)**
- [ ] Implement comprehensive logging for all actions (see Layer 5)
- [ ] Create `backend/services/analytics.py` for behavioral profiling:
  - **User Profiling:** Login patterns, API call frequency, typical code languages
  - **Entity Profiling:** Agent health metrics, resource usage baselines
  - **Relationship Mapping:** User-to-code submissions, access patterns
- [ ] Define baseline for each user (e.g., "User X typically submits Python code 9-5 EST")
- [ ] Calculate anomaly scores using statistical methods (z-score, IQR)
- [ ] Flag anomalies: 
  - Unusual login location/time
  - Bulk data downloads
  - Access to unrelated resources
  - Code patterns indicating exfiltration attempts

**Phase 4B: Security Information & Event Management (SIEM)**
- [ ] Deploy Elastic Stack (Elasticsearch + Logstash + Kibana) or Grafana Loki
- [ ] Ship all logs to centralized SIEM with structured JSON format
- [ ] Create dashboards for:
  - Authentication events (success/failure rates)
  - Data access patterns
  - Error rates by service
  - Top users by activity
- [ ] Set up alerts for:
  - >5 failed logins in 5 minutes
  - Access attempts outside business hours
  - Unusual API endpoint access patterns
  - Large code submissions (>500KB)

**Phase 4C: Machine Learning Anomaly Detection** (Advanced)
- [ ] Use scikit-learn for unsupervised anomaly detection (Isolation Forest, One-Class SVM)
- [ ] Collect training data (normal behavior over 30 days)
- [ ] Implement features: 
  - Request timing, payload size, endpoint frequency
  - User resource usage patterns
  - Cross-service call chains
- [ ] Deploy model as separate service or use cloud ML (Azure Sentinel, GCP Chronicle)
- [ ] Retrain monthly with recent data

**Phase 4D: Real-Time Alerting & Response**
- [ ] Create alerting pipeline (webhook to PagerDuty/Slack/Email)
- [ ] Implement automated containment actions:
  - Temporary lockout after suspicious activity
  - Require password reset after anomaly
  - Session termination for anomalous behavior
- [ ] Tiered alert severity:
  - **P0 (Critical):** Compromised credentials detected → immediate response
  - **P1 (High):** Unusual admin activity → 15min response
  - **P2 (Medium):** Unusual user behavior → 1hr review
  - **P3 (Low):** Policy violations → daily review
- [ ] Integrate with incident management (Jira, ServiceNow)

**Phase 4E: Threat Intelligence Enrichment**
- [ ] Subscribe to threat intelligence feeds (MISP, AlienVault OTX)
- [ ] Cross-reference suspicious IPs/user agents with known bad actors
- [ ] Block known malicious IPs at firewall
- [ ] Integrate with VirusTotal/Hybrid Analysis for uploaded code scanning

### **LAYER 5: INTEGRITY & AUDITING**

#### Goals
- Immutable, tamper-evident logging
- Complete audit trail for every data access and admin action
- Non-repudiation and forensic readiness
- Automated compliance reporting

#### Implementation Steps

**Phase 5A: Structured Audit Logging**
- [ ] Define canonical audit log schema (JSON):
```json
{
  "timestamp": "2025-05-10T14:30:00Z",
  "event_id": "uuidv4",
  "actor": {"user_id": "...", "ip": "...", "user_agent": "..."},
  "action": "code.analyze",
  "resource": {"type": "code_submission", "id": "uuid"},
  "outcome": "success/failure",
  "context": {"language": "python", "size_bytes": 1234},
  "session_id": "uuid",
  "trace_id": "uuid"
}
```
- [ ] Create `backend/utils/audit.py` with `audit_log()` function
- [ ] Replace print/log statements with structured audit logging
- [ ] Ensure all service calls generate audit events
- [ ] Include request metadata: IP, user agent, geolocation (optional)

**Phase 5B: Immutable Log Storage**
- [ ] Write logs to append-only files (no overwrites)
- [ ] Use write-once-read-many (WORM) storage or S3 Object Lock (cloud)
- [ ] Hash each log entry and chain using Merkle tree (optional advanced)
- [ ] Sign log batches with private key for tamper detection
- [ ] Daily log rotation with gzip compression
- [ ] Off-site log archival (retain 7 years for compliance)
- [ ] Tools: `rotatingfilehandler` with `when='midnight'`, backup to cloud

**Phase 5C: Real-Time Auditing & Dashboards**
- [ ] Build audit dashboard (React + FastAPI endpoint):
  - Recent activity feed
  - Failed authentication attempts map
  - Data access by user/time
  - Admin action history
- [ ] Implement `/audit` endpoint with filtering by:
  - Date range
  - User ID
  - Action type
  - Resource
- [ ] Add export to CSV/PDF for compliance reports
- [ ] Schedule weekly compliance reports (GDPR, SOC2, HIPAA checklists)

**Phase 5D: Integrity Verification**
- [ ] Generate SHA-256 hashes for all stored data
- [ ] Create periodic integrity checks (daily):
  - Recalculate hashes, compare to stored
  - Alert on mismatches (potential tampering)
- [ ] Implement digital signatures for critical data:
  - Sign history.json with Ed25519 private key
  - Verify signature on each read
  - Store public key in secure location
- [ ] Enable file integrity monitoring (FIM) on critical files
- [ ] Backups must also be signed and verified

**Phase 5E: Forensic Snapshots & Retention**
- [ ] Daily snapshots of entire data store (encrypted)
- [ ] Retention policy:
  - Audit logs: 7 years (immutable)
  - Analysis history: 2 years (active), 5 years (archived)
  - Raw code submissions: 90 days (then delete, keep redacted metadata)
- [ ] Immutable backup storage ( Glacier, cold archive)
- [ ] Document chain of custody for forensic investigations
- [ ] Annual retention policy review

**Phase 5F: Security Event Timeline**
- [ ] Create unified timeline view correlating:
  - Auth events
  - Code submissions
  - Admin actions
  - System errors
  - Network events
- [ ] Index by user, resource, time
- [ ] Export incident reports in standard formats (STIX/TAXII)

## 8. OPERATIONAL PROCEDURES

### Key Management SOP
1. Generate master key in HSM or KMS
2. Distribute to 3 key custodians (shamir secret sharing optional)
3. Rotate keys annually via automated pipeline
4. Test key rotation quarterly in staging
5. Document key escrow recovery process

### Incident Response Playbook
**Scenario 1: Compromised User Account**
1. Detect via UEBA anomaly alert (P0 severity)
2. Auto-lock user account
3. Notify security team via PagerDuty
4. Require MFA re-enrollment + password reset
5. Review all activity from past 30 days
6. Check for lateral movement or data exfiltration
7. Document in incident tracker

**Scenario 2: Malicious Insider**
1. Detect via audit log of unusual admin activity
2. Freeze all user sessions immediately
3. Revoke all active tokens
4. Preserve forensic snapshot of all systems
5. Escalate to senior management + legal
6. Initiate investigation with chain of custody
7. Restore from clean backups within 1 hour

**Scenario 3: Data Breach**
1. Contain: isolate affected systems
2. Assess: determine scope of exposed data
3. Notify: legal, compliance, affected users (GDPR 72h)
4. Remediate: patch vulnerabilities, rotate keys
5. Review: conduct post-mortem, update controls
6. Report: file with authorities if required

### Compliance & Auditing
- Quarterly internal audit of access controls
- Annual penetration test by third party
- Monthly review of audit logs for anomalies
- Bi-annual key rotation verification
- Real-time compliance dashboards for SOC2, ISO 27001

## 9. RISK MITIGATION & TRADE-OFFS

### Recognized Trade-offs
| Benefit | Cost/Complexity | Mitigation |
|---------|----------------|------------|
| Strong encryption | Performance overhead (~5-10%) | Use hardware acceleration (AES-NI) |
| MFA for all users | User friction | Phased rollout + training |
| Micro-segmentation | Operational complexity | Infrastructure as Code + automation |
| Audit logging storage | Cost (logs grow 50GB/mo) | Compression + tiered storage (hot/warm/cold) |
| ML anomaly detection | False positives | Tune thresholds + human-in-the-loop review |

### Residual Risks
- **Supply Chain:** Dependency vulnerabilities → Use SCA tools (Dependabot/Snyk)
- **Zero-Day Exploits:** Unpatched vulnerabilities → Harden systems + EDR
- **Insider with Valid Credentials:** UEBA will detect anomalous behavior
- **Cryptographic Breakthrough:** Use algorithm agility to migrate
